Deadlocked

by: Thomas Snyder

This game was made for my level 2 project at The League of Amazing Programmers.
Click here to download and play!

1. Why I Made This Game

I liked the idea of a randomized game that still had an element of strategy to it. I also liked the idea because it wasn't just a generic platformer or top-down.

2. What I Learned

I learned about a lot of exploits and errors in the game that I changed, and the best feedback I got was about the difficulty of tracking your stats, which lead to me adding an updating stat section as the title of the main map.

3. Architecture of My Game

So the map holds the tiles, which are decided by a randomizer. Each tile contains a different set of actions, and the player can move to adjacent tiles. Picking up weapons and magic items will help you fight the goblins, and the gold and treasures you get can buy you more things from the trader. The game was designed around and build upon a tile array and within that, the battle frame which contains the details and the interaction in combat. The Deadlocked class sets up the game and runs all the other classes as it contains the action performed. The tiles the player can interact with to run the action performed are defined by the player class, and the Encounters class randomizes the tiles, defines what they do, and holds the information and execution of combat.

4. Technical Implementation

I'm proud of the trader as he was a constant source of bugs and problems within the game. I had to create a tile that had an interactive menu for the player to navigate and it had to be set so that even if the player clicked "cancel" it didn't duplicate the trader. Additionally, I had to reset the randomized trader location after every shopping encounter and make sure it doesn't land on top of other spaces. Afterward, I had to make sure the icon and coloration transferred correctly. Finally, I had to make sure there weren't any complications with the shopping, include pressing the "Ok" or "Cancel" buttons at an inopportune time

5. Most Challenging Part

The most challenging part of the project was the bugfixing as it was strenuous, repetitive, and generally annoying. After thinking I was finished, I kept getting pelt with new issues. Then, those issues would have to be fixed, which either caused new issues or didn't solve the current issue at all. In the end, I restructured a majority of the game's code to prevent commonly occurring bugs, for example, the checks for goblins remaining and for setting new trader location as to avoid duplication.

6. Future Fixes and Updates

In the future I'd like to streamline the shopping process because right now it's completely text based and having to buy more than one item gets repetitive. Also, sometimes you mistype the item you want, and you have to reset the whole process. I would improve this by mapping it all to buttons, have a magic items, weapons, and equipment buttons that when clicked would lead to their respective menus and have buttons to decide the purchase.
